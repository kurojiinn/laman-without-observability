# Laman — Статус проекта

> Сервис универсальной доставки, Грозный. Дата: апрель 2026.

---

## 1. Что реализовано

### Backend (Go + Gin)

| Модуль | Что есть |
|--------|----------|
| **Auth** | OTP через SMS.RU, JWT (HS256, 24ч), регистрация/вход по телефону, роли (CLIENT / COURIER / PICKER / ADMIN) |
| **Catalog** | Категории, подкатегории, магазины (6 типов), товары, поиск и фильтрация, изображения |
| **Orders** | Создание заказов (гостевые + авторизованные), статусная машина, история заказов, расчёт стоимости (5% сбор + 200₽ доставка) |
| **Payments** | Таблица платежей, методы CASH / TRANSFER |
| **Courier** | Трекинг локации через Redis GEO, смены, поиск ближайшего курьера (5км → 10км → алерт) |
| **Picker** | Авторизация по phone+password, список заказов магазина, обновление статусов, SSE-нотификации |
| **Favorites** | Добавление/удаление/просмотр избранных товаров |
| **Admin** | Статистика, управление магазинами и товарами, массовый импорт CSV/Excel, статусы заказов |
| **Observability** | Zap логи, OpenTelemetry + Jaeger трейсинг, Prometheus метрики, Grafana дашборд, Telegram уведомления |

### Frontend

| Приложение | Статус |
|------------|--------|
| **iOS (SwiftUI)** | Каталог, корзина, оформление заказа, профиль, история, авторизация, Keychain |
| **Client Web (Next.js)** | Каталог с фильтрами, корзина, оформление, профиль, история, избранное, авторизация |
| **Picker Panel (React/Vite)** | Логин, очередь заказов, детали заказа, обновление статусов, SSE real-time |
| **Admin Panel (React/Vite)** | Магазины, товары, импорт, заказы, статистика |

### Инфраструктура

- Docker Compose: api, postgres, redis, jaeger, prometheus, grafana, admin-panel, picker-panel
- GitLab CI/CD: lint → build → deploy (staging auto, prod manual)
- Goose миграции: 14 файлов, применяются автоматически при старте
- Kubernetes манифесты (laman-dev / laman-staging namespaces)

## 2. Уязвимости и критические проблемы

### Критические

**JWT секрет не ротируется**
- Один секрет на всё, хранится в `.env`
- При компрометации все токены нужно инвалидировать вручную перезапуском сервиса
- Нет blacklist токенов — выход из аккаунта не инвалидирует токен

**Пароли пикеров — bcrypt не подтверждён**
- В коде есть `password_hash` в БД, но явного использования bcrypt не видно
- Если хранится plaintext — критическая уязвимость

**Admin Basic Auth по HTTP**
- Пароль передаётся в base64 (не шифровании) при каждом запросе
- Без HTTPS — полностью открыт для перехвата
- Дефолты `admin/admin` в `.env` — легко угадать

**OTP код — 4 цифры**
- 10 000 вариантов, нет лимита попыток на verify-code endpoint
- Брутфорс за секунды: достаточно 10к запросов

**Секреты в `.env` рядом с кодом**
- `JWT_SECRET`, `SMS_RU_KEY`, `TG_BOT_TOKEN` в одном файле
- Риск случайного коммита (хотя `.gitignore` должен закрывать)

### Высокие

**Нет rate limiting на API**
- Кроме SMS cooldown (60с) — больше нигде нет ограничений
- Возможен DDoS, брутфорс, спам заказами

**pprof открыт без авторизации**
- `/debug/pprof/*` доступен всем без авторизации
- Раскрывает информацию о внутреннем состоянии сервиса

**CORS Origins включает локальные адреса в проде**
- В `.env` сейчас одновременно localhost и prod IP
- При деплое нужно разделять конфиги dev/prod

**Загрузка файлов без валидации**
- `/uploads` — статика отдаётся напрямую
- Нет проверки типа файла при импорте CSV/Excel (возможна загрузка исполняемых файлов)

**SMS код выводится в лог в plaintext**
- `[OTP] phone=... code=1234` — в production логах не должно быть OTP кодов

### Средние

**JWT без refresh токена**
- Токен живёт 24 часа, после истечения — полный re-login
- Плохой UX: пользователь теряет сессию без предупреждения

**Нет пагинации на некоторых эндпоинтах**
- GET /orders, GET /catalog/products — при росте данных будут тормозить

**Guest заказы не верифицированы**
- Гость может указать любой номер телефона без подтверждения
- Возможен спам заказами с чужими номерами

## 3. Что нужно поправить (приоритет)

### Срочно (до первых реальных пользователей)

- [ ] **Rate limiting** — добавить middleware на все публичные эндпоинты (особенно auth)
- [ ] **Лимит попыток OTP** — максимум 5 попыток, потом блокировка на 15 минут
- [ ] **Увеличить OTP до 6 цифр** — 1 000 000 вариантов вместо 10 000
- [ ] **Убрать OTP из production логов** — только в dev/debug режиме
- [ ] **Закрыть pprof** — только для внутренней сети или совсем убрать
- [ ] **HTTPS** — TLS через nginx reverse proxy или cert-manager в k8s
- [ ] **Проверить bcrypt** для паролей пикеров

### Важно (в ближайший спринт)

- [ ] **Refresh токены** — короткий access (15мин) + долгий refresh (30 дней)
- [ ] **Blacklist JWT** — при logout инвалидировать токен через Redis
- [ ] **Разделить конфиги** — dev.env / staging.env / prod.env
- [ ] **Пагинация** на всех list-эндпоинтах (limit/offset или cursor)
- [ ] **Верификация гостевого телефона** — OTP перед созданием заказа
- [ ] **Валидация файлов** при загрузке — MIME-type, размер, расширение

### Технический долг

- [ ] **Тесты** — покрытие критической бизнес-логики (сервисный слой)
- [ ] **Документация API** — Swagger/OpenAPI
- [ ] **Healthcheck детальный** — проверка БД и Redis в /health
- [ ] **Graceful shutdown** — дождаться завершения in-flight запросов (уже есть, проверить timeout)
- [ ] **Индексы** — проверить EXPLAIN ANALYZE на часто используемых запросах
