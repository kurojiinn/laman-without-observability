Общая архитектура

01

Зачем вообще нужна эта архитектура?

Когда приложение живёт только на локальной машине, всё просто: открыл терминал, запустил Go-сервер, открыл браузер. Но на продакшн-сервере у тебя одновременно работают три фронтенда, бэкенд, база данных и хранилище файлов. Нужно решить несколько задач сразу:

  • Безопасность — база данных не должна быть доступна из интернета. Только твой бэкенд должен к ней подключаться.
  • Маршрутизация — три разных домена (laman.app, admin.laman.app, picker.laman.app) должны вести на разные сервисы, но все они живут на одном сервере с одним IP.
  • HTTPS — браузеры требуют зашифрованного соединения. Кто-то должен заниматься SSL-сертификатами.
  • Изоляция — если один сервис упал, остальные должны продолжать работать.

Решение всех этих задач — поставить перед всеми сервисами reverse proxy (в нашем случае Nginx). Он принимает все входящие запросы, разбирается куда их направить, завершает SSL-соединение и перенаправляет трафик к нужному сервису уже внутри защищённой сети.

Что такое reverse proxy и почему Nginx?

Обычный прокси (forward proxy) — это посредник между клиентом и интернетом. Reverse proxy — это посредник между интернетом и твоими серверами. Пользователь видит только Nginx, а куда идут его запросы — он не знает.

Nginx выбирают по трём причинам: он очень быстрый при работе со статикой и проксировании (основан на event-loop, не создаёт поток на каждое соединение), надёжный (используется LinkedIn, Airbnb, Netflix), и его конфигурация читабельна. Для нашего масштаба любой nginx-сервер справится с нагрузкой без малейших усилий.

Сервер: подготовка

02

Какой сервер выбрать?

Для первого деплоя подойдёт любой VPS (Virtual Private Server) — это виртуальная машина в дата-центре. Популярные провайдеры: DigitalOcean, Hetzner (дешевле всего в Европе), Timeweb Cloud или Selectel (если нужно в России). Hetzner — оптимальное соотношение цены и качества: 2 vCPU + 2 GB RAM за ~€4/месяц.

Выбирай Ubuntu 22.04 LTS — это самая стабильная версия с долгосрочной поддержкой до 2027 года. Большинство туториалов и официальных инструкций написаны под неё.

Насчёт мощности: для начала хватит 1 vCPU + 1 GB RAM. Go-бэкенд очень экономный, Nginx вообще почти не потребляет ресурсы. Основной потребитель — это Next.js при серверном рендеринге и PostgreSQL. Когда пойдут реальные пользователи — можно апгрейдить сервер без пересоздания (большинство провайдеров позволяют менять тариф онлайн).

Почему Docker, а не просто запустить на сервере?

Можно просто скопировать бинарник Go на сервер и запустить. Это работает. Но тогда ты столкнёшься с проблемами: нужно вручную управлять версиями Node.js, PostgreSQL, Redis — и на каждом сервере настраивать всё заново. Обновление одного сервиса может сломать другой.

Docker решает это через контейнеризацию: каждый сервис живёт в своём изолированном окружении со всеми зависимостями. Docker Compose позволяет описать все сервисы в одном файле и запустить командой docker compose up -d. Это воспроизводимо: то, что работает у тебя локально, будет работать на сервере один в один.

Docker Compose

03

Как работают Docker-сети

Когда ты запускаешь docker compose up, Docker создаёт виртуальную сеть (в нашем случае internal) и подключает к ней все сервисы. Внутри этой сети каждый контейнер доступен по имени сервиса из docker-compose.yml.

Например: Go-бэкенд не знает ничего про IP-адрес базы данных. Он просто обращается к хосту postgres — и Docker сам разрешает это имя в реальный IP контейнера. Поэтому в .env пишем DB_HOST=postgres, а не localhost или какой-то IP. Это магия Docker DNS.

Ключевое правило: директива ports: в docker-compose пробрасывает порт наружу (на хост-машину и в интернет). Если ты не хочешь чтобы PostgreSQL был доступен из интернета — просто не пиши ports: для него. Внутри сети он всё равно будет доступен для бэкенда.

volumes: — где хранятся данные

По умолчанию данные внутри Docker-контейнера исчезают при перезапуске. Для PostgreSQL и Redis это катастрофа. Docker volumes решают это: данные сохраняются на диске хост-машины и переживают перезапуски контейнеров. postgres_data:/var/lib/postgresql/data означает «примонтировать именованный volume postgres_data в папку /var/lib/postgresql/data внутри контейнера».

💡 Все сервисы в одной internal сети. Nginx — единственный с ports:. Остальные сервисы общаются по имени контейнера.
yaml — docker-compose.yml (production) 
services:

  postgres:
    image: postgres:15-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB:       ${DB_NAME}
      POSTGRES_USER:     ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks: [internal]
    # Нет ports: — снаружи недоступен

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis_data:/data
    networks: [internal]

  api:
    build:
      context: ./Backend
    restart: unless-stopped
    env_file: .env
    volumes:
      - ./Backend/uploads:/app/uploads
    networks: [internal]
    depends_on: [postgres, redis]

  client-app:
    build:
      context: ./Frontend/client-app
      dockerfile: Dockerfile.prod
    restart: unless-stopped
    networks: [internal]

  admin-panel:
    build:
      context: ./Frontend/admin-panel
      dockerfile: Dockerfile.prod
    networks: [internal]

  picker-panel:
    build:
      context: ./Frontend/picker-panel
      dockerfile: Dockerfile.prod
    networks: [internal]

  nginx:
    image: nginx:alpine
    restart: unless-stopped
    ports:
      - "80:80"    # ← единственный открытый наружу
      - "443:443"
    volumes:
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - /etc/letsencrypt:/etc/letsencrypt:ro
    networks: [internal]

  minio:
    image: minio/minio
    restart: unless-stopped
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER:     ${MINIO_USER}
      MINIO_ROOT_PASSWORD: ${MINIO_PASSWORD}
    volumes:
      - minio_data:/data
    networks: [internal]

volumes:
  postgres_data:
  redis_data:
  minio_data:

networks:
  internal:
    driver: bridge

Порты и сетевая схема

04

Что такое порт и почему важно их контролировать

Порт — это числовой адрес, по которому можно достучаться до конкретного сервиса на машине. У одной машины один IP-адрес, но 65535 портов. Браузер по умолчанию обращается на порт 80 (HTTP) или 443 (HTTPS). PostgreSQL слушает 5432, Redis — 6379.

Если ты откроешь порт 5432 наружу — любой человек в интернете сможет попытаться подключиться к твоей базе данных напрямую, подбирая пароль. Это называется brute force attack, и боты занимаются этим круглосуточно автоматически. Именно поэтому firewall (ufw) закрывает все порты кроме 22 (SSH), 80 и 443.

Внутри Docker-сети порты не имеют отношения к портам хост-машины. Контейнеры общаются напрямую по внутренним адресам — это быстро и безопасно.

Dockerfile для фронтендов

05

Multi-stage build — почему это важно

Обычный Dockerfile для Node.js-приложения содержит: Node.js, npm, все dev-зависимости (prettier, eslint, typescript и ещё тысяча пакетов в node_modules). Итоговый образ весит 800 MB — 1.5 GB. Это медленно скачивается, медленно запускается и является угрозой безопасности — в образе лежат инструменты разработки, которые не нужны в проде.

Multi-stage build решает это элегантно: используется два образа. Первый (builder) — тяжёлый, со всеми зависимостями — только для сборки. Второй (runner) — лёгкий, берёт из builder только то, что нужно для запуска: собранные файлы. Для Vite это вообще просто папка dist/ со статическими HTML/CSS/JS. Образ получается 20–50 MB вместо гигабайта.

Чем отличается Next.js от Vite в контексте деплоя

Vite (admin-panel, picker-panel) — это чистый клиентский SPA. После npm run build получается папка dist/ с HTML, CSS и JS файлами. Это статика — её может раздавать любой веб-сервер, в том числе nginx-alpine без Node.js вообще. Плюс: минимальный образ, нет серверного процесса.

Next.js (client-app) — может работать в двух режимах: как статика (export) или как серверный Node.js процесс (SSR/SSG). В режиме output: 'standalone' Next.js собирает минимальный Node.js-сервер с только нужными файлами. Вес: ~100 MB вместо 1 GB+. Для серверного рендеринга и API routes это необходимо.

⚠️ Текущие Dockerfile запускают npm run dev — это режим разработки, НЕ для продакшна. Нужны Dockerfile.prod с настоящим билдом.
Next.js (client-app)
Vite (admin / picker)
dockerfile — Dockerfile.prod 
# ── Стадия 1: сборка (тяжёлый образ, ~800MB) ─
FROM node:20-alpine AS builder
WORKDIR /app

COPY package*.json ./
RUN npm ci --frozen-lockfile
COPY . .

# Build-time переменные (не секреты, встраиваются в JS-бандл)
ARG NEXT_PUBLIC_API_URL
ENV NEXT_PUBLIC_API_URL=$NEXT_PUBLIC_API_URL
RUN npm run build

# ── Стадия 2: запуск (лёгкий образ, ~100MB) ──
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

# Берём только собранные файлы из builder
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

EXPOSE 3000
CMD ["node", "server.js"]
💡 В next.config.ts обязательно добавить output: 'standalone' иначе server.js не будет создан.

Nginx — конфигурация

06

Как Nginx понимает, куда направить запрос

Когда браузер открывает https://laman.app/catalog, запрос приходит на IP сервера. Nginx смотрит на заголовок Host в запросе (там будет laman.app) и сравнивает с директивой server_name в своих конфигах. Нашёл совпадение — применяет этот блок и делает proxy_pass к нужному контейнеру.

Это называется виртуальные хосты: один сервер с одним IP обслуживает несколько доменов, просто смотря на заголовок запроса. Именно поэтому ты можешь иметь laman.app, api.laman.app и admin.laman.app — все на одном сервере.

SSE и WebSocket — особый случай

Обычные HTTP-запросы короткие: клиент спросил — сервер ответил — соединение закрылось. SSE (Server-Sent Events) и WebSocket — это долгоживущие соединения, которые остаются открытыми минуты или часы. Сборщик в picker-panel слушает SSE-поток о новых заказах — это соединение живёт пока сборщик работает.

По умолчанию Nginx через 60 секунд закрывает «молчащие» соединения. Для SSE нужно поднять proxy_read_timeout до 3600s (1 час). Также нужны заголовки Upgrade и Connection для WebSocket — без них соединение не обновится до WS-протокола.

nginx — conf.d/api.conf 
# HTTP → всегда редиректим на HTTPS
server {
    listen 80;
    server_name api.laman.app;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name api.laman.app;

    ssl_certificate     /etc/letsencrypt/live/api.laman.app/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.laman.app/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    location / {
        proxy_pass         http://api:8080;
        proxy_http_version 1.1;

        # Для WebSocket и SSE (долгие соединения)
        proxy_set_header Upgrade           $http_upgrade;
        proxy_set_header Connection         "upgrade";
        proxy_set_header Host               $host;
        proxy_set_header X-Real-IP          $remote_addr;
        proxy_set_header X-Forwarded-Proto  $scheme;

        # Важно для SSE — иначе nginx закроет через 60 сек
        proxy_read_timeout  3600s;
        proxy_send_timeout  3600s;
    }

    # MinIO файлы (картинки товаров) через Nginx
    location /files/ {
        proxy_pass http://minio:9000/laman-uploads/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}

HTTPS через Let's Encrypt

07

Как работает HTTPS и зачем он нужен

HTTP — это открытый текст. Всё что ты отправляешь: пароли, токены, личные данные — проходит через интернет в читаемом виде. Любой, кто контролирует сеть между тобой и сервером (провайдер, wifi-роутер в кафе), может это прочитать или подменить.

HTTPS = HTTP + TLS. TLS (Transport Layer Security) шифрует всё соединение. Браузер и сервер договариваются о ключах шифрования так, что перехватить трафик бессмысленно — без ключей это просто случайные байты. Для этого нужен SSL-сертификат — цифровой документ, подтверждающий что сервер действительно является laman.app, а не кем-то другим.

Let's Encrypt — это бесплатный центр сертификации, который автоматически выдаёт и обновляет SSL-сертификаты. До него сертификаты стоили $100–300 в год. Сейчас это бесплатно и автоматизировано. Сертификат действует 90 дней — certbot обновляет его автоматически.

Certbot или Traefik?

Certbot устанавливается на сервер и работает независимо от Docker. Он выпрашивает сертификат у Let's Encrypt, доказывая владение доменом через HTTP-challenge (Let's Encrypt обращается к твоему серверу по HTTP и проверяет специальный файл). Сертификаты хранятся в /etc/letsencrypt/ и пробрасываются в контейнер nginx. Надёжно, проверено, большинство хостингов так и делают.

Traefik — это прокси нового поколения. Он сам следит за Docker-контейнерами, читает их labels и автоматически настраивает маршруты и сертификаты. Не нужно писать конфиги nginx вручную — только добавить labels к сервису. Минус: немного сложнее для дебага, меньше контроля над нюансами. Для небольшого проекта как Laman — хороший выбор.

Docker labels → сам настраивает роуты и SSL. Нулевая конфигурация nginx. Сертификаты обновляются автоматически.

# Получить сертификат для каждого домена

  -d laman.app -d www.laman.app \

sudo certbot certonly --webroot -w /var/www/certbot -d admin.laman.app ...

PostgreSQL

08

Как не потерять данные

Данные PostgreSQL хранятся в Docker volume (postgres_data). Это значит они выживают при рестарте контейнера. Но если ты удалишь контейнер командой docker compose down -v (флаг -v удаляет volumes) — всё пропадёт. Поэтому нужны регулярные бэкапы.

Стратегия бэкапов для старта: ежедневный дамп командой pg_dump, сохраняемый в отдельную папку. Хранить последние 7 дней (старше автоматически удалять). Этого достаточно для MVP. Когда пойдут реальные деньги — подключить внешнее хранилище (S3/rclone для автоматической синхронизации бэкапов).

Важно: бэкапы на том же сервере — не настоящие бэкапы. Если сервер сгорит, потеряешь всё. Для серьёзного продакшна нужно копировать дампы на другой сервер или в облако.

✅ Внутри Docker-сети хост БД — это имя сервиса: DB_HOST=postgres, а не localhost.
bash — бэкапы и восстановление 
# Ручной бэкап
docker compose exec postgres pg_dump -U postgres laman > backup_$(date +%Y%m%d).sql

# Crontab: ежедневно в 2:00
0 2 * * * docker compose exec -T postgres pg_dump -U postgres laman \
  | gzip > /backups/laman_$(date +\%Y\%m\%d).sql.gz

# Удалять бэкапы старше 7 дней
0 4 * * * find /backups -name "*.sql.gz" -mtime +7 -delete

# Восстановление из бэкапа
docker compose exec -T postgres psql -U postgres laman < backup.sql

# Запустить миграции после деплоя
docker compose exec api make migrate-up

Redis

09

Зачем Redis если есть PostgreSQL

Redis — это база данных в оперативной памяти (in-memory). Чтение/запись в 100 раз быстрее чем в PostgreSQL, потому что данные не на диске, а в RAM. За это платишь тем, что данных меньше помещается и они могут пропасть при выключении (если не настроить persistence).

В Laman Redis используется для трёх задач: хранение OTP-кодов (живут 5 минут и удаляются автоматически — Redis умеет TTL из коробки), GPS-координаты курьеров (Geo-команды Redis — встроенный поиск ближайших точек), и кэш JWT-сессий.

Параметр --appendonly yes включает AOF persistence: каждая операция записывается в файл. При рестарте Redis воспроизводит этот файл и восстанавливает состояние. Это замедляет запись на ~10%, но зато данные не теряются при перезапуске контейнера.

yaml — redis в docker-compose с persistence 
redis:
  image: redis:7-alpine
  restart: unless-stopped
  command: >
    redis-server
    --requirepass ${REDIS_PASSWORD}
    --appendonly yes           # сохранять данные при рестарте
    --appendfsync everysec     # сбрасывать на диск каждую секунду
  volumes:
    - redis_data:/data
  networks: [internal]

Хранилище файлов (S3 / MinIO)

10

Почему нельзя просто хранить файлы на диске сервера

Сейчас картинки товаров лежат в Backend/uploads/. Это работает, но у этого подхода есть проблемы:

  • Нет резервирования — если сервер упадёт, картинки пропадут вместе с данными.
  • Сложно масштабировать — если запустить второй экземпляр бэкенда, у него не будет доступа к файлам первого.
  • Занимает дисковое место сервера — SSD на VPS дорогой, а файлы растут быстро.

S3 (Simple Storage Service) — стандарт хранения файлов в облаке от Amazon. Объектное хранилище: загружаешь файл по HTTP, получаешь публичную URL, раздаёшь клиентам. MinIO — это open-source реализация S3 API, которую можно запустить в Docker на своём сервере. Код бэкенда при этом одинаков — что для AWS S3, что для MinIO.

Как это работает для Laman

Когда администратор загружает картинку товара — бэкенд отправляет файл в MinIO через S3 API. MinIO хранит его в своём volume. Когда клиент открывает страницу товара — он запрашивает https://api.laman.app/files/products/image.jpg. Nginx проксирует этот запрос в MinIO и возвращает файл.

Для клиента это выглядит как обычный файл по URL. Но на самом деле он хранится в объектном хранилище с управлением версиями, доступом и возможностью переехать на AWS S3 одной строкой конфига.

⚠️ Текущий код бэкенда сохраняет файлы на диск. Нужно добавить S3-клиент — пример ниже.

go — загрузка файла в S3 
// go get github.com/aws/aws-sdk-go-v2/service/s3
// Работает одинаково для AWS S3 и MinIO
func UploadImage(ctx context.Context, file multipart.File, filename string) (string, error) {
    cfg, _ := config.LoadDefaultConfig(ctx,
        config.WithCredentialsProvider(
            credentials.NewStaticCredentialsProvider(
                os.Getenv("S3_ACCESS_KEY"),
                os.Getenv("S3_SECRET_KEY"), ""),
        ),
        config.WithRegion(os.Getenv("S3_REGION")),
    )
    client := s3.NewFromConfig(cfg)

    _, err := client.PutObject(ctx, &s3.PutObjectInput{
        Bucket:      aws.String(os.Getenv("S3_BUCKET")),
        Key:         aws.String("products/" + filename),
        Body:        file,
        ContentType: aws.String("image/jpeg"),
    })
    if err != nil { return "", err }
    return os.Getenv("S3_PUBLIC_URL") + "/products/" + filename, nil
}

Переменные окружения

11

Почему переменные окружения, а не конфиг-файлы

12-factor app — это методология разработки SaaS-приложений. Один из её принципов: конфигурация хранится в окружении, а не в коде. Почему? Потому что код попадает в git, а секреты в git попадать не должны. Переменные окружения позволяют одному и тому же Docker-образу работать и как dev, и как prod — просто с разными .env файлами.

.env файл должен быть в .gitignore. Вместо него в репозиторий кладут .env.example — шаблон с примерами значений без реальных секретов.

🚨 .env никогда не должен попадать в git! Проверить историю: git log --all --full-history -- .env
.env — production 
# ── База данных ─────────────────────────────
DB_HOST=postgres        # имя контейнера, не localhost!
DB_PORT=5432
DB_PASSWORD=<openssl rand -base64 32>
DB_NAME=laman

# ── Сервер ──────────────────────────────────
PUBLIC_URL=https://api.laman.app   # домен, не IP!
JWT_SECRET=<openssl rand -hex 32>

# ── Критично для прода ──────────────────────
SMS_RU_TEST=false     # ← ОБЯЗАТЕЛЬНО false!
COOKIE_SECURE=true   # ← ОБЯЗАТЕЛЬНО true!

# ── CORS — только HTTPS домены ──────────────
CORS_ORIGINS=https://laman.app,https://admin.laman.app,https://picker.laman.app

# ── Redis ───────────────────────────────────
REDIS_PASSWORD=<openssl rand -base64 32>
REDIS_URL=redis://:${REDIS_PASSWORD}@redis:6379

# ── MinIO / S3 ──────────────────────────────
S3_ENDPOINT=http://minio:9000
S3_BUCKET=laman-uploads
S3_PUBLIC_URL=https://api.laman.app/files

CI/CD

12

Что такое CI/CD и зачем он нужен

CI (Continuous Integration) — автоматическая проверка кода при каждом пуше: запуск линтера, тестов, сборки. CD (Continuous Delivery/Deployment) — автоматический деплой после успешной проверки.

Без CI/CD деплой выглядит так: SSH на сервер, git pull, docker compose build, docker compose up. Это можно забыть сделать, сделать неправильно или задеплоить непроверенный код. С CI/CD: сделал push в main — GitLab запустил линтер, собрал образы, задеплоил. Всё автоматически, воспроизводимо, с историей запусков.

В нашем pipeline деплой в продакшн — ручной шаг (when: manual). Это значит: код автоматически проверяется и собирается, но на прод уходит только когда ты нажмёшь кнопку в GitLab UI. Это защита от случайного деплоя сломанного кода.

yaml — .gitlab-ci.yml 
stages: [lint, build, deploy_prod]

variables:
  REGISTRY: registry.gitlab.com/khadzhiev404/laman-backend
  IMAGE_TAG: $CI_COMMIT_SHORT_SHA  # уникальный тег = хэш коммита

lint:
  stage: lint
  script:
    - cd Backend && golangci-lint run ./...

build:
  stage: build
  script:
    - docker build -t $REGISTRY/api:$IMAGE_TAG ./Backend && docker push $REGISTRY/api:$IMAGE_TAG
    - docker build -t $REGISTRY/client:$IMAGE_TAG \
        --build-arg NEXT_PUBLIC_API_URL=https://api.laman.app \
        -f ./Frontend/client-app/Dockerfile.prod ./Frontend/client-app
    - docker push $REGISTRY/client:$IMAGE_TAG

deploy_prod:
  stage: deploy_prod
  when: manual   # нажать кнопку в GitLab UI — защита от случайного деплоя
  script:
    - ssh deploy@server "cd /app && docker compose pull && docker compose up -d"
    - ssh deploy@server "docker compose exec -T api make migrate-up"
